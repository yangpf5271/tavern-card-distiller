# 讯飞闲鱼统计

## 触发词
/讯飞闲鱼统计、API销售统计、查看API Key

## 功能说明
管理闲鱼API Key销售情况，每个Key最多5个名额。

### 价格与名额规则
- 一个Key成本：199元（一次性付款）
- 一个Key可卖：5份（基础名额）
- ⭐ **超发规则：每个Key最多超发1份，即最多6份**（5基础 + 1超发）
- 每份售价：160元
- 盈利计算：销售额 - Key成本（每个Key一次性减199元）

### 盈利统计
- 按Key统计：每个Key的已售份数、销售额、Key成本、盈利
- 按日期统计：每日销售额、Key成本、盈利
- 汇总统计：总销售额、总Key成本、总盈利、退款、推流成本、净盈利

### 从Key7开始计入成本和盈利（Key6不计入）

## 数据文件
- 销售数据：`./skills/xunfei-xianyu-tracker/apikeys.md`
- 退款记录：`./skills/xunfei-xianyu-tracker/refunds.md`
- Key备份：`./skills/xunfei-xianyu-tracker/key-backup.md` ⚠️ 这是备份，不是可售卖的Key！
- 推流成本：`./skills/xunfei-xianyu-tracker/promotion-cost.md`

## ⚠️ 重要原则

1. **数据一致性**：任何对 Key 的增删改操作，都必须同步更新本地文件！内存操作 = 文件操作，两者必须同步！
2. **退款处理**：退款只记录退款历史（refunds.md），不删除买家，但要释放该Key的名额（已使用-1，剩余+1）
3. **退款记录**：保存在 refunds.md，可查询退款历史
4. **推流成本**：记录在 promotion-cost.md
5. **Key备份**：保存在 key-backup.md

## 初始化
首次使用需要告诉技能：
- 有几个API Key
- 每个Key的序号
- 已卖出几个

## 支持的操作

### 1. 初始化/增加Key
告诉技能：`我有X个API Key，序号分别是XXX，已卖出N个`
格式自由，技能会解析并更新数据文件

### 2. 新卖出一个API
告诉技能：`用XXX这个key，卖给了咸鱼商家"商家名"`
技能会：
- 找到对应key
- 检查买家名字是否重复或相似（跨Key检查）
- 无重复/相似 → 直接添加买家记录
- 有重复/相似 → **先判断时间是否超过一个月**：
  - 超过一个月 → ✅ **自动认定为复购**，无需询问，直接添加并标注 `🔁 Key-XX复购`
  - 一个月内 → ⚠️ 显示提醒并询问确认（"检测到买家名'xxx'与Key-XX的'xxx'重复，购买时间MM-DD，距今X天，是否确认添加？"），必须用户确认后才添加
- **⚠️ 购买时间必须用 `date +%m-%d\\ %H:%M` 获取系统真实时间，不要自己编造或推算时间！**
- 更新剩余名额
- 回复当前统计

### 买家名字重复检测规则
- 检查所有Key的所有买家列表
- 包含相同字符串 → 重复
- 包含相似字符串（模糊匹配）→ 相似
- 提醒示例："检测到买家名'xxx'与'xxx'重复/相似，是否确认添加？"
- 必须用户明确确认后才能添加

### ⭐ 重复买家自动复购认定规则（新增）
- **时间阈值：一个月**。如果检测到的重复买家，其**最近一次**购买时间距今超过一个月（30天），则自动认定为**复购**，无需弹确认，直接添加并标注 `🔁 Key-XX复购`
- **如何判断**：提取该买家所有历史购买时间，取最新的那条，与当前系统日期比较，超过30天 → 自动复购
- **一个月内**的重复仍然需要弹确认（可能是误操作、迁移、或短期复购，需人工判断）
- **时间解析**：购买时间格式为 `MM-DD HH:MM`（如 `05-07 21:24`），解析时假定年份为当前年份（跨年情况：如当前是1月而购买时间是12月，则年份为去年）

### 3. 查看统计
直接问：
- "现在有几个key？"
- "用了多少名额？"
- "还剩多少？"

### 4. 特定Key查询
问：`查看XXX这个key的买家`
技能会列出该key的所有买家记录

### 5. 增删改买家
- 添加买家：`给XXX这个key添加买家"商家名"`
- 删除买家：`删除XXX这个key的买家"商家名"`
- 修改买家：`把XXX这个key的买家"旧名字"改成"新名字"`

### 6. 退款处理 ⭐
告诉技能：`XXX这个key的买家"商家名"退款了`

技能会执行以下操作：
1. 释放名额：已使用减1，剩余加1
2. 更新该Key的盈利统计（销售额减160元）
3. 将退款记录写入 `refunds.md` 退款历史文件
4. 更新总统计中的退款汇总（退款笔数、退款金额、净盈利）
5. **同步更新本地文件 apikeys.md 和 refunds.md**

**注意：退款只记录，不删除买家！但必须释放名额！**

**退款记录格式**（写入refunds.md）：
```markdown
## 退款记录

| 序号 | Key | 买家 | 退款日期 | 退款金额 |
|------|-----|------|----------|----------|
| 1 | Key-7 | Forrest | 2026-04-23 | 160元 |
```

### 7. 查看退款历史 ⭐
直接问：
- "查看退款记录"
- "有哪些退款？"
- "退款历史"

技能会读取 `refunds.md` 并展示所有退款记录，包括：
- 退款总数
- 退款总金额
- 每笔退款的详细信息（Key、买家、时间、金额）

### 8. 退款统计汇总
退款记录会自动计入总统计：
- 累计盈利 = 正常销售额 - Key成本
- **退款：X笔共Y元**
- **推流成本：Z元**
- **净盈利 = 累计盈利 - 退款金额 - 推流成本**

### 9. 月度/周期战绩统计 ⚠️ 必须含Key成本！

告诉技能：`这个月战绩` / `本月统计` / `5月卖了多少钱` 等

技能会：
1. 提取指定月份的所有买家记录（按购买时间筛选）
2. 统计该月销量份数和销售额
3. **⚠️ 必须提取该月新创建的Key，计算其成本！**（通过创建时间字段筛选）
4. 统计该月推流成本和退款金额
5. **净利润 = 销售额 - 当月新Key成本 - 推流成本 - 退款**
6. 列出当月各Key明细（销量、回本状态等）

**⚠️ 高频错误：只算销售额减推流减退款，忘了Key成本！**
Key成本是实打实的进货钱，不能忽略。4月创建的Key成本已在4月计入，5月只算5月新创建的Key成本。

**月度统计检查清单：**
- [ ] 销售额（份数 × 160）
- [ ] 当月新Key成本（创建时间在该月的Key × 199元）
- [ ] 当月推流成本
- [ ] 当月退款金额
- [ ] 净利润 = 以上四项加减

### 10. 推流成本记录
告诉技能：`今日推流成本X元`
技能会：
- 记录到 promotion-cost.md
- 更新总统计中的推流成本汇总
- 同步更新 apikeys.md 的汇总表

### 11. 新增Key登记 ⭐
告诉技能：`新增 keyXX [备注文字] [API Key]`
也可以在新增Key的同时卖出首单：`keyXX 卖一个，买家XXX`（先登记再卖）

技能会：
1. 获取系统时间作为创建时间（`date +%Y-%m-%d`）
2. 在 `## 盈利汇总` 之前插入新 Key 的完整信息块（序号、创建时间、Key成本199元、API Key、备注、名额上限5、已使用0、剩余5、状态✅可售、空买家列表、盈利统计初始值-199元）
3. 如果同时有首单买家，一并添加（已使用1、剩余4、盈利-39元等）
4. 在汇总表中插入新 Key 行（在合计行之前）
5. 更新合计行：Key成本+199、已售+N、销售额+160N、盈利+(160N-199)
6. 更新总计区域：**总Key数+1、总名额+5**、已使用+N、累计已售+N、累计销售额+160N、累计盈利+(160N-199)、净盈利、ROI、利润率
7. **⚠️ ROI 的分母变了**：新 Key 增加了总 Key 成本，ROI = 净盈利 / (新总Key成本 + 推流成本) × 100，必须重新计算！

**新增Key时的摘要更新检查清单**：
- [ ] 汇总表插入 Key-XX 行
- [ ] 合计行：Key成本、已售、销售额、盈利 四列全部更新
- [ ] 总计：总Key数、总名额
- [ ] 总计：已使用、累计已售、累计销售额、累计盈利
- [ ] 总计：净盈利、ROI、利润率（三个都因分母/分子变化需重算）

## ⚠️ 常见陷阱

1. **购买时间必须取系统真实时间**：每次添加买家时，必须调用 `date +%m-%d\ %H:%M` 获取当前系统时间作为购买时间。绝对不能自己编造时间（如按消息顺序递增分钟数），否则会导致时间记录完全失真，影响按日期统计和退款查询。
2. **购买时间格式必须用24小时制**：时间格式为 `MM-DD HH:MM`（如 `04-28 18:33`），必须使用24小时制，避免AM/PM混淆。不要省略日期只写 `HH:MM`。磊哥哥明确要求。
3. **汇总表要同步更新**：每次增删改买家后，必须同步更新 apikeys.md 中的按Key汇总表、合计行、以及总计区域的所有数字（已使用、累计已售、累计销售额、累计盈利、净盈利、ROI、利润率）。
4. **退款时买家名必须精确匹配**：用户说退款时，如果买家名在系统中找不到，要列出所有买家让用户确认，不要猜测。
5. **超发Key退款的特殊处理**：如果退款的Key处于超发状态（如已超发1份共6份），退款后名额恢复为正常上限（如恢复为5份），Key状态从"已售罄（可超发）"变为"已售罄"。即：退款释放超发名额，优先取消超发状态。
6. **非标准退款金额**：退款不一定都是160元，可能是扣除了使用费的金额（如125元）。处理方式一样，只是退款金额不同。要确认用户说的金额。
7. **迁移买家到其他Key**：用户可能要求将买家从一个Key移到另一个Key。操作包括：①源Key删除买家，释放名额（超发数的减少），更新已售份数/销售额/盈利；②目标Key追加买家，增加超发数，更新已售份数/销售额/盈利；③汇总表同步更新两行；④合计和总计不变（内部调换）。
8. **不要用 `read_file` 的 offset/limit 读取后直接操作**：read_file 输出带行号，直接操作可能导致文件内容被行号污染。始终用 `terminal` 直接读取和写入原始文件。
9. **编辑后务必验证**：每次修改后用以下命令验证关键区域是否更新正确：
   - 详情区：`awk '/^## Key-XX/,/^## Key-YY/' apikeys.md | head -20`（推荐，比 `sed -n 'N,Mp'` 可靠）
   - 汇总表：`grep 'Key-XX |' apikeys.md`
   - 注意交叉对比详情区和汇总表的数值（已售份数、销售额、盈利）是否一致！
10. **⚠️ 盈利行的精确匹配陷阱**：不同 Key 可能拥有相同的盈利值（如 Key-20 和 Key-21 都是 `441元`），Python 脚本中的 `replace('盈利：441元', '盈利：601元')` 会匹配到多个 Key 导致错误更新。解决方案：①在 Python 脚本中用 Key 范围限定（只在 `key_start` 到 `key_end` 之间查找替换）；②或用 `patch` 工具加更多上下文（如包含 `Key成本：199元` 行 + 空行 + `## Key-XX` 标题行）来精确唯一匹配；③编辑后必须验证 Key 区域和汇总表两个地方的盈利数字是否一致。
11. **⚠️ `lines.insert()` + `for` 循环 = 索引错位（高频坑！）**：在 `for i in range(start, end)` 循环内使用 `lines.insert(i+1, ...)` 插入买家行后，后续所有行的索引 +1，但 `for` 循环继续按原索引迭代，导致后续的 `if` 条件匹配到错误的行（或直接跳过）。**现象**：盈利行、销售额行经常更新失败，尤其是多个 Key 的盈利值相同时（如 Key-20 和 Key-21 都有 `盈利：441元`）。**解决方案**：
   - **方案A（推荐）**：先收集所有修改，循环结束后统一应用。不要在循环内 `insert`。
   - **方案B**：插入后用 `break` 退出循环，剩余修改用 `patch` 工具单独处理。
   - **方案C**：倒序迭代（`for i in range(end-1, start-1, -1)`），插入时索引不受影响。
   - **方案D**：插入买家行后，立即 `i += 1` 手动跳过刚插入的行（需要改用 `while` 循环）。
   - **最推荐的做法**：把买家添加和数值更新分成两个独立步骤——先找到插入位置并 `insert`，然后重新遍历 Range 做数值替换（此时索引已稳定）。
12. **⚠️ Range 越界（`list index out of range`）**：`insert` 之后文件变长，原来硬编码的 `range(key_start, key_start + N)` 中 `N` 可能不够或 `key_start + N` 超出 `len(lines)`。**解决方案**：所有 range 一律用 `range(key_start, min(key_start + N, len(lines)))`，尤其在插入行后重新遍历时。
13. **⚠️ 半写入崩溃的连锁污染（高频坑！）**：脚本如果在 `insert` + `write` 之后、数值替换之前崩溃（如 Range 越界），文件已被部分修改。重跑脚本时会再次 `insert` 同一买家，产生**重复行**。**解决方案**：①插入前先检查买家是否已存在（`if buyer_name in ''.join(lines[key_start:])`），存在则跳过 insert 只做数值更新；②或者把所有修改收集完后**一次性写入**，不要中途 write——中途 write 是万恶之源。

14. **⚠️ heredoc 特殊字符转义陷阱**：当买家名包含 `*`、`&`、`$`、`\\`` 等 shell 特殊字符时（如 `x***3`），使用 `python3 << 'PYEOF'` heredoc 会报错 `"Foreground command uses '&' backgrounding"` ——即使用了单引号分隔符，`&` 仍会被 shell 解析为后台运行符号。**解决方案**：不要用 heredoc，改用 `write_file` 将 Python 脚本写入临时文件（如 `/tmp/update_xxx.py`），再 `terminal` 执行 `python3 /tmp/update_xxx.py`。同理，买家名中的 `$` 也会被 shell 展开、反引号会被当作命令替换，所以凡是买家名含特殊字符，一律走文件方式。

15. **⚠️ `in` 子串匹配导致缩进丢失（高频坑！）**：盈利统计的子项有2空格缩进（`  - 已售份数：1`），用 `if '已售份数：1' in line` 虽然能匹配到，但 `replace('已售份数：1', '已售份数：2')` 只替换文本不保留缩进，导致缩进丢失、格式错乱。**现象**：替换后 `  - 已售份数：2` 变成 `- 已售份数：2`（丢失2空格前缀）。**解决方案**：①用 `line.strip().startswith('- 已售份数：')` 做匹配判断，然后用 `line.replace('旧值', '新值')`（保留原始格式）；②或在 replace 中包含完整原文本（如 `'  - 已售份数：1'` 包含缩进）；③编辑后务必用 `grep -A5 'Key-XX'` 检查缩进一致性。

16. **⚠️ `read_file` 默认 500 行截断导致库存漏算**：apikeys.md 文件已超过 500 行（当前约560行），`read_file` 默认只返回前500行，排在后面的 Key（如 Key-23 在552行）会被截断。**现象**：查看库存时漏掉了末尾的 Key，剩余名额统计偏少。**解决方案**：①查看库存/搜索买家时优先用 `terminal` + `grep`（如 `grep '剩余.*[1-9]'`），速度快且无截断；②如果必须用 `read_file`，指定足够大的 `limit` 或分页读取；③用 `execute_code` 内 `open().readlines()` 直接读原始文件（不经过 read_file 工具）。

17. **⚠️ 详情区与汇总表不同步（高频坑！）**：脚本执行后，`## Key-XX` 详情区的数值（已售份数、销售额、盈利）可能与下方汇总表的同一 Key 行不一致。比如详情区显示 已售份数:6/销售额:960元 但汇总表正确显示 4/640元。**现象**：脚本中 `replace` 匹配到了 Key 范围外的其他行，或 Key 的详情区在之前操作中已被污染。**解决方案**：①编辑完成后必须同时验证详情区和汇总表；②用 `awk '/^## Key-XX/,/^## Key-YY/' apikeys.md | head -N` 提取单个 Key 完整详情区验证（比 `sed -n 'N,Mp'` 可靠，后者在多行表达式时易报 `unknown command` 错误）；③数值用 `grep 'Key-XX |' apikeys.md` 交叉对比汇总表行。

19. **⚠️ `'|' not in line` 过滤导致总计区漏更新（高频坑！）**：总计区字段前缀**不统一**——"已使用"的前缀是 `- `（如 `- 已使用：190`），而累计已售/累计销售额/累计盈利/净盈利/ROI/利润率等字段的前缀是 `| `。如果用 `'|' not in line` 做过滤，"已使用"行会因不带 `|` 被放过，但其他字段被跳过。**现象**：添加买家后，Key 详情区的已使用被意外多 +1，总计区的已使用却没变。**解决方案**：①更新总计区时**不要**用 `'|' not in line` 或 `'|' in line` 前缀过滤；②统一用精确值匹配（如 `'已使用：190' in line`）并限定在 `### 总计` 后的 range 内；③编辑后务必 `grep '已使用：' apikeys.md` 检查总计区和各 Key 详情区数值是否一致。

## 🔧 文件编辑实操指南（重要！）

### 为什么不能用 patch/sed
- `patch` 工具的模糊匹配在 apikeys.md 上频繁失败——因为所有 Key 的汇总表行 `| Key-X | 199元 | N | XXX元 | XXX元 |` 结构高度相似，匹配不到唯一结果
- `sed` 对中文有正则转义问题（元、已售罄、已超发等字符），部分表达式会报 `Invalid preceding regular expression`
- **推荐方法**：用 `terminal` + Python heredoc 脚本来编辑文件

### 推荐的文件编辑流程

#### ⚠️ 禁止使用 heredoc！统一用文件方式
由于买家名可能包含 `*`、`&`、`$` 等 shell 特殊字符，**禁止**使用 `python3 << 'PYEOF'` heredoc。统一流程：① `write_file` 写入 `/tmp/xxx.py` → ② `terminal` 执行 `python3 /tmp/xxx.py`。

⚠️ **每次编辑前先检测行号污染**（见陷阱23）：`grep -c '|[0-9][0-9]*|## Key' apikeys.md`，如果 >0 先执行清理脚本。
